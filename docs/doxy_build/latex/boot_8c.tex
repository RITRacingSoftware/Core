\doxysubsection{Detailed Description}
Core bootloader. 

This core library component implements a bootloader that allows boards to be programmed over CAN.\hypertarget{boot_8c_autotoc_md2}{}\doxysubsubsection{\texorpdfstring{Theory of operation}{Theory of operation}}\label{boot_8c_autotoc_md2}
The STM32\+G473 has 512k of FLASH, which is split into two banks of 256k each. The chip can be configured to boot either from the first or the second bank by configuring the non-\/volatile {\ttfamily BFB2} bit in the option byte registers. When the {\ttfamily BFB2} bit is set in the FLASH option byte register, the boots from the second bank, otherwise, it boots from the first bank. However, there is also an option to temporarily swap the banks and run the code in the second bank even when booting from the first bank. This allows the code in the non-\/booting bank to be verified before finalizing the swap. If the verification fails, then the chip will fall back to the working code in the booting bank.

Programming a board takes place according to the following process\+:
\begin{DoxyEnumerate}
\item The programmer sends a command to the target board to enter the bootloader
\item The programmer sends program data to the target board. The target board writes the program data to the non-\/booting bank
\item After each block of data is written, the target board reads the block back so the programmer can verify it
\item Once all of the data has been written, the programmer commands the target board to switch to the non-\/booting bank
\item The target board board resets and performs a soft bank swap
\item The programmer commands the target board to enter the bootloader in the non-\/booting bank
\item The programmer sends a command to the bootloader in the non-\/booting bank to verify that FDCAN communication is working
\item The programmer commands the target board to binalize the bank swap
\item The target board updates the option byte, resets, and runs the new code
\end{DoxyEnumerate}

The bootloader keeps track of its state across resets and between banks using the {\ttfamily boot\+\_\+state} variable, which is stored at the highest RAM address (above the stack). This variable is not initialized when the chip is reset, so its value is always preserved unless the chip is power cycled. The highest 24 bits of {\ttfamily boot\+\_\+state} are known as the boot key and must always be set to {\ttfamily 0x\+ABCDEF}. If the boot key is incorrect, an error is raised. This might occur if the {\ttfamily boot\+\_\+state} variable is not correctly configured.\hypertarget{boot_8c_autotoc_md3}{}\doxysubsubsection{\texorpdfstring{FDCAN format}{FDCAN format}}\label{boot_8c_autotoc_md3}
The bootloader communicates with the programmer board using FDCAN with extended IDs. The extra bits in the ID are used to communicate they type of message and the address to be programmed (if required), so all 64 bytes in the body of the message can be used for data.

The bootloader consists of four main components\+: the startup script, the boot state machine, the \doxylink{boot_8c_a17df2565b3ec70a6a540f71a38bc4cdc}{core\+\_\+boot\+\_\+init()} function, and an entry point.\hypertarget{boot_8c_autotoc_md5}{}\doxysubsubsubsection{\texorpdfstring{Startup script}{Startup script}}\label{boot_8c_autotoc_md5}
For the bootloader to work, the default startup script must be replaced by the startup script startup\+\_\+stm32g473xx.\+s. The startup script defines the interrupt handlers for the program, including the reset handler. By default, the reset handler initializes the stack and jumps to the application code. The modified startup script required for the bootloader also runs the boot state machine before performing any additional initialization.\hypertarget{boot_8c_autotoc_md6}{}\doxysubsubsubsection{\texorpdfstring{Boot state machine}{Boot state machine}}\label{boot_8c_autotoc_md6}
The boot state machine runs before any other code and updates the boot state or soft swaps as necessary. The BSM first checks the nature of the reset. If the reset was caused externally (by pulling the n\+RST pin low), then the state is reset to NORMAL and the BSM continues to the application code.

The behavior of the BSM depends on whether it is running in the booting or non-\/booting banks. In the booting bank, the BSM will change the state to VERIFY\+\_\+\+SOFT\+\_\+\+SWAP and soft-\/swap if the state is VERIFY and continue to application code otherwise.

In the non-\/booting bank, the BSM will change the state to SOFT\+\_\+\+SWITCHED if the state is VERIFY\+\_\+\+SOFT\+\_\+\+SWAP. Otherwise, the BSM will set the NB\+\_\+\+ERROR bit and reset. This causes the chip to return to the booting bank.

The \doxylink{boot_8c_a17df2565b3ec70a6a540f71a38bc4cdc}{core\+\_\+boot\+\_\+init()} function is called by application code after the clocks, GPIO, and FDCAN modules needed for the bootloader have been initialized. This function initializes the RX filter for the board\textquotesingle{}s bootloader ID. The function also checks the boot state and enters the bootloader if necessary. See the state diagram above for details.\hypertarget{boot_8c_autotoc_md8}{}\doxysubsubsubsection{\texorpdfstring{Entry point}{Entry point}}\label{boot_8c_autotoc_md8}
The software enters the bootloader by calling \doxylink{boot_8c_ab2be0dfac3cabc71734c82636d313d87}{core\+\_\+boot\+\_\+reset\+\_\+and\+\_\+enter()}. The FDCAN RX interrupt hander in \doxylink{can_8c}{can.\+c} will automatically call this function if it receives a packet addressed to its bootloader FDCAN ID containing the command to enter the bootloader
